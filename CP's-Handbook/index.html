    CP's Handbook

    Notes and suggestions for Competitive Programmer's Handbook.

    Greedy algorithms¶

    • Say that if coins are divisors of each other then the greedy always works correctly.
    • Deadlines problem is very confusing. It's intuitive that you get 0 points if you don't finish before a deadline. Those artificial deadlines just increase the score by d_i for each task. If we need to choose a subset of tasks and finish each of them before its deadline, then we should still sort by duration and then do dp[i][time] - best possible score so far.
    • Is "minimizing sums" greedy?

    Dynamic programming¶

    • Say that INF is something like 1e9+5 because for INF=INT_MAX it overflows when we increase it by 1 here: best = min(best, solve(x-c)+1);
    • "and has lower constant factors" -> "and has lower constant factor" (?)
    • It might be hard for some people to move from recursive to iterative dp.
    • "Counting the number of solutions" - explain what happens when we don't care about the order of used coins. Explain an issue of double counting.
    • Maybe change 'm' to 'mod' when explaining that sometimes we need to compute the value modulo something?
    • For LIS, you can mention an alternative solution with dp[k] - the smallest value of the last element of an increasing subsequence of length k. This is also O(N^2) but can be easily improved to O(N*log(N)).
    • It is obvious that we can't reuse elements in the following quote? Maybe it's better to say we have N items and we can choose some subset of them. "Given a list of weights [w1,w2,...,wn], determine all sums that can be constructed using the weights."
    • For "Counting tilings", mention the "broken profile" technique.

    Amortized analysis¶

    • Give some code/pseudocode for two pointers.
    • Mention that the binary search solution for 2SUM requires sorting too. Just change "In such a solution, we iterate through the array" to "... through the sorted array".

    Range queries¶

    • Segment trees are better than BIT because they can solve more problems (minimum; intervals that are not prefixes). So it might be better to explain them instead of BIT first. Basically, you don't really have to know BIT.
    • I like the drawing on page 88 (98 in PDF), that shows which intervals in BIT are affected by a change.
    • Explain how to initialize BIT and a segment tree.
    • Range updates can be done not only for sums and we don't need an array of differences for that. Just do the same thing as previously but the update function will do something with interval (in the same way as previously described).

    Bit manipulation¶

    • In "Set implementation", change int x to unsigned int x. Overflowing a signed variable is UB. (?)
    • Explain bitsets for "Counting subgrids".
